## 二、安全性问题

### 2.1 🔴 严重：SQL 注入漏洞

**位置**：`traffic/traffic.go` 第 170-179 行，`buildWhereClause` 函数

```go
func buildWhereClause(sinceStr string, filter FilterParams) string {
    where := fmt.Sprintf("created_at >= '%s'", sinceStr)
    if filter.ModelID > 0 {
        where += fmt.Sprintf(" AND model_id = %d", filter.ModelID)
    }
    // ...
}
```

**问题分析**：`sinceStr` 通过 `time.Time.Format()` 生成，虽然当前调用路径是安全的，但 `buildWhereClause` 作为公共函数，其参数 `sinceStr` 是字符串类型而非 `time.Time`，未来其他开发者调用时可能直接传入用户输入。`fmt.Sprintf` 拼接 SQL 是典型的 SQL 注入风险模式。

**影响范围**：`GetDashboardStats`、`GetModelRanking`、`GetProviderRanking`、`GetUserStats`、`GetUserTrafficRecords` 所有统计查询函数。

**优化建议**：

```go
// 使用参数化查询替代字符串拼接
func buildWhereClause(since time.Time, filter FilterParams) (string, []interface{}) {
    where := "created_at >= ?"
    args := []interface{}{since}
    if filter.ModelID > 0 {
        where += " AND model_id = ?"
        args = append(args, filter.ModelID)
    }
    if filter.ProviderID > 0 {
        where += " AND provider_id = ?"
        args = append(args, filter.ProviderID)
    }
    return where, args
}
```

同时在 `GetUserTrafficRecords` 中 `user_id` 也是直接拼接进 SQL（第 351 行），需一并修复。

### 2.2 🔴 严重：供应商 API Key 明文泄露

**位置**：`admin/admin.go` 第 540-543 行

```go
func (s *Server) handleListProviders(c *gin.Context) {
    var providers []database.Provider
    s.db.Find(&providers)
    c.JSON(200, providers)  // 直接返回包含 API Key 的完整对象
}
```

**问题分析**：`Provider` 结构体的 `APIKey` 字段虽然有 `json:"api_key"` 标签，但在返回给前端时没有任何脱敏处理。管理员查看供应商列表时，所有供应商的 API Key 以明文形式传输到浏览器，存在以下风险：

1. **传输泄露**：即使使用 HTTPS，浏览器开发者工具、日志、缓存仍可能记录
2. **XSS 攻击面**：前端存储的 Token 若被窃取，可通过供应商接口获取所有上游 API Key
3. **内部威胁**：低权限管理员也能看到所有供应商的 API Key

**优化建议**：

```go
func (s *Server) handleListProviders(c *gin.Context) {
    var providers []database.Provider
    s.db.Find(&providers)

    var result []gin.H
    for _, p := range providers {
        result = append(result, gin.H{
            "id":          p.ID,
            "name":        p.Name,
            "description": p.Description,
            "base_url":    p.BaseURL,
            "api_key":     maskAPIKey(p.APIKey), // 脱敏: "sk-abc...xyz"
            "timeout":     p.Timeout,
            "retry":       p.Retry,
            "status":      p.Status,
            "created_at":  p.CreatedAt,
            "updated_at":  p.UpdatedAt,
        })
    }
    c.JSON(200, result)
}

func maskAPIKey(key string) string {
    if len(key) <= 8 {
        return "****"
    }
    return key[:4] + "..." + key[len(key)-4:]
}
```

### 2.3 🔴 严重：CORS 配置过度开放

**位置**：`admin/admin.go` 第 48-55 行

```go
router.Use(cors.New(cors.Config{
    AllowOrigins:     []string{"*"},
    AllowCredentials: true,
    // ...
}))
```

**问题分析**：`AllowOrigins: ["*"]` 与 `AllowCredentials: true` 同时使用是 CORS 规范中的冲突配置。浏览器在收到此响应时会拒绝携带凭据的请求（因为通配符源不允许携带凭据），但实际上某些浏览器的实现可能不一致，导致安全行为不可预测。更严重的是，`*` 允许任何网站发起对管理 API 的跨域请求，为 CSRF 攻击打开了大门。

**优化建议**：

```go
router.Use(cors.New(cors.Config{
    AllowOrigins:     []string{"http://localhost:5173", "https://your-domain.com"},
    AllowMethods:     []string{"GET", "POST", "PUT", "DELETE", "OPTIONS"},
    AllowHeaders:     []string{"Origin", "Content-Type", "Authorization"},
    AllowCredentials: true,
    MaxAge:           12 * time.Hour,
}))
```

### 2.4 🟡 中等：管理员更新接口无字段白名单

**位置**：`admin/admin.go` 第 572-588 行

```go
func (s *Server) handleUpdateProvider(c *gin.Context) {
    var req map[string]interface{}
    c.ShouldBindJSON(&req)
    delete(req, "id")
    delete(req, "created_at")
    s.db.Model(&database.Provider{}).Where("id = ?", id).Updates(req)
}
```

**问题分析**：使用 `map[string]interface{}` 接收更新数据，仅删除了 `id` 和 `created_at`，攻击者可以注入任意字段，例如修改 `updated_at` 时间戳，或在未来新增字段时意外写入不应被修改的数据。`handleUpdateModel` 也有同样问题。

**优化建议**：使用结构体绑定代替 `map[string]interface{}`，或显式定义允许更新的字段白名单：

```go
type UpdateProviderReq struct {
    Name        *string `json:"name"`
    Description *string `json:"description"`
    BaseURL     *string `json:"base_url"`
    APIKey      *string `json:"api_key"`
    Timeout     *int    `json:"timeout"`
    Retry       *int    `json:"retry"`
    Status      *string `json:"status"`
}
```

### 2.5 🟡 中等：JWT 密钥管理不安全

**位置**：`server/main.go` 第 66-74 行，`config.yaml` 第 25 行

```go
jwtSecret := []byte(cfg.JWTSecret)
if len(jwtSecret) == 0 || string(jwtSecret) == "" {
    secretBytes := make([]byte, 32)
    rand.Read(secretBytes)
    jwtSecret = secretBytes
}
```

**问题分析**：

1. 配置文件中硬编码了 JWT 密钥 `f7f34a1e50c2758091197b5004df6e460bf8c28ddd6204382017afc28eae90ba`，若代码仓库泄露，所有 Token 将被伪造
2. 自动生成的密钥仅在内存中，重启后旧 Token 失效，生产环境不友好
3. `rand.Read` 使用的是 `crypto/rand`，密钥质量本身没有问题，但缺乏密钥轮换机制

**优化建议**：

1. 从环境变量读取 JWT 密钥，不写入配置文件
2. 支持密钥轮换：维护新旧两个密钥，旧密钥仅用于验证
3. 配置文件示例中移除实际密钥值

```go
jwtSecret := os.Getenv("JWT_SECRET")
if jwtSecret == "" {
    jwtSecret = cfg.JWTSecret
}
```

### 2.7 🟡 中等：API Key 提取方式不安全

**位置**：`proxy/proxy.go` 第 366-375 行

```go
func extractAPIKey(r *http.Request) string {
    apiKey := r.Header.Get("Authorization")
    if apiKey == "" {
        apiKey = r.URL.Query().Get("key")  // URL 参数传递 API Key
    }
    // ...
}
```

**问题分析**：支持通过 URL 查询参数 `?key=xxx` 传递 API Key，这会导致：

1. API Key 被记录在服务器访问日志中
2. API Key 出现在浏览器历史记录中
3. API Key 被嵌入 Referrer 头发送到外部站点
4. URL 更容易被中间人攻击截获

**优化建议**：移除 URL 参数传递 API Key 的方式，仅支持 `Authorization: Bearer` 头：

```go
func extractAPIKey(r *http.Request) string {
    apiKey := r.Header.Get("Authorization")
    if apiKey == "" {
        return ""
    }
    return strings.TrimPrefix(apiKey, "Bearer ")
}
```

### 2.8 🟡 中等：JSON 解析模型名存在绕过风险

**位置**：`proxy/proxy.go` 第 410-439 行

```go
func extractModelFromJSON(data []byte) string {
    str := string(data)
    idx := strings.Index(str, `"model"`)
    // 手动解析 JSON ...
}
```

**问题分析**：手动解析 JSON 而非使用标准库 `encoding/json`，存在以下问题：

1. 无法处理嵌套对象中的 `"model"` 字段（如 `{"data": {"model": "xxx"}}`）
2. 无法处理 JSON 注释（虽然标准 JSON 不支持注释，但部分 LLM API 可能容忍）
3. 无法处理 Unicode 转义（如 `\u006dodel` 等同于 `model`）
4. 无法处理重复键（JSON 规范未定义行为）

**优化建议**：

```go
func extractModelFromJSON(data []byte) string {
    var body struct {
        Model string `json:"model"`
    }
    if err := json.Unmarshal(data, &body); err != nil {
        return ""
    }
    return body.Model
}
```

### 2.9 🟢 低风险：代理请求缺少请求体大小限制

**位置**：`proxy/proxy.go` 第 136-142 行

```go
bodyBytes, err := io.ReadAll(r.Body)
```

### 2.10 🟢 低风险：缺少速率限制

**问题分析**：代理服务器和管理服务器均未实现速率限制，攻击者可以：

1. 暴力破解 API Key

## 三、数据泄露风险

### 3.1 🔴 严重：供应商 API Key 明文存储

**位置**：`database/database.go` Provider 结构体

```go
type Provider struct {
    // ...
    APIKey string `gorm:"column:api_key;size:500" json:"api_key"`
    // ...
}
```

**问题分析**：供应商的上游 API Key 以明文存储在数据库中，一旦数据库被入侵（SQL 注入、备份泄露、内部人员），所有供应商的 API Key 将全部泄露。这些 Key 通常关联着高额的付费账户，泄露后果严重。

**优化建议**：

1. 使用 AES-256-GCM 加密存储 API Key，密钥从环境变量获取
2. 返回给前端时始终脱敏
3. 数据库备份中的 API Key 应为密文

```go
// 加密存储
func EncryptAPIKey(plaintext, encryptionKey string) (string, error) {
    block, _ := aes.NewCipher([]byte(encryptionKey))
    gcm, _ := cipher.NewGCM(block)
    nonce := make([]byte, gcm.NonceSize())
    rand.Read(nonce)
    ciphertext := gcm.Seal(nonce, nonce, []byte(plaintext), nil)
    return base64.StdEncoding.EncodeToString(ciphertext), nil
}
```

### 3.3 🟡 中等：供应商 API Key 通过缓存传播

**位置**：`cache/cache.go` 第 118-128 行

```go
info := ModelProviderInfo{
    ProviderID:        mp.ProviderID,
    ProviderName:      provider.Name,
    ProviderModelName: mp.ProviderModelName,
    BaseURL:           provider.BaseURL,
    APIKey:            provider.APIKey,  // API Key 存储在内存缓存中
    // ...
```

**问题分析**：供应商的 API Key 被加载到内存缓存中，在以下场景存在风险：

1. 内存转储（crash dump）中可见
2. 调试工具可读取进程内存
3. 缓存对象在日志中打印时可能泄露

**优化建议**：代理层确实需要使用 API Key 来发起请求，无法完全避免内存中存储。但应：

2. 在缓存结构体中将 `APIKey` 字段标记为敏感，使用 `fmt.Stringer` 接口实现脱敏输出
3. 考虑使用加密内存区（如 `memguard`）

### 3.5 🟢 低风险：前端 localStorage 存储 Token

**位置**：`web/src/stores/user.ts`

```typescript
localStorage.setItem('token', data.token)
localStorage.setItem('role', data.user.role)
```

**问题分析**：JWT Token 存储在 localStorage 中，易受 XSS 攻击窃取。虽然使用 localStorage 比 Cookie 更容易防范 CSRF，但一旦存在 XSS 漏洞，攻击者可以立即获取 Token。

**优化建议**：

1. 确保所有用户输入经过严格转义，防止 XSS
2. 考虑使用 httpOnly Cookie + CSRF Token 的方案
3. 设置较短的 Token 过期时间（当前为 24 小时，可考虑缩短至 2-4 小时，配合 Refresh Token）

## 四、性能问题

### 4.1 🟡 中等：数据库查询缺少分页和索引优化

**位置**：`admin/admin.go` 多处

```go
// 全表扫描，无分页
func (s *Server) handleListUsers(c *gin.Context) {

func (s *Server) handleListProviders(c *gin.Context) {
```

**问题分析**：所有列表查询均无分页和字段选择，当数据量增长时会：

2. 内存占用高（将整张表加载到 Go 对象）
3. 网络传输量大（JSON 序列化所有字段）

**优化建议**：

```go
func (s *Server) handleListUsers(c *gin.Context) {
    page, pageSize := s.getPage(c), s.getPageSize(c)
    var users []database.User
    var total int64
    s.db.Model(&database.User{}).Count(&total)
    s.db.Offset((page-1)*pageSize).Limit(pageSize).Find(&users)
    // 返回分页结果
}
```

### 4.2 🟡 中等：代理层无 HTTP 连接池

**位置**：`proxy/proxy.go` 第 182-183 行

```go
client := &http.Client{Timeout: timeout}
resp, err = client.Do(proxyReq)
```

**问题分析**：每次代理请求都创建新的 `http.Client`，导致：

1. 每次请求都建立新的 TCP 连接（无连接复用）
2. TLS 握手开销大（每次请求都需重新握手）
3. 连接数不受控制，可能导致端口耗尽

**优化建议**：

```go
// 使用全局连接池
var proxyClient = &http.Client{
    Timeout: 60 * time.Second,
    Transport: &http.Transport{
        MaxIdleConns:        100,
        MaxIdleConnsPerHost: 20,
        IdleConnTimeout:     90 * time.Second,
        DisableKeepAlives:   false,
    },
}
```

### 4.3 🟡 中等：缓存重载未做防抖

**位置**：`admin/admin.go` 多处

```go
go s.cache.Reload()  // 每次数据变更后异步重载
```

**问题分析**：每次数据变更（创建、更新、删除）都触发 `cache.Reload()`，如果短时间内多次操作（如批量创建映射），会触发多次全量重载。`Reload` 函数持有写锁，期间所有读操作阻塞。

**优化建议**：

```go
// 添加防抖机制
type DebouncedCache struct {
    cache   *Cache
    timer   *time.Timer
    mu      sync.Mutex
    delay   time.Duration
}

func (d *DebouncedCache) Reload() {
    d.mu.Lock()
    defer d.mu.Unlock()
    if d.timer != nil {
        d.timer.Stop()
    }
    d.timer = time.AfterFunc(d.delay, func() {
        d.cache.Reload()
    })
}
```

### 4.4 🟡 中等：流量统计查询跨月分表性能差

**位置**：`traffic/traffic.go` 各统计函数

```go
func GetDashboardStats(db *gorm.DB, since time.Time, filter FilterParams) (DashboardStats, error) {
    tables, err := database.GetTrafficTables(db)  // 获取所有分表
    // 构建所有分表的 UNION ALL 查询
}
```

**问题分析**：

1. `GetTrafficTables` 每次都查询 `sqlite_master`，结果应缓存
2. 当分表数量增多（如运行 2 年后有 24 张分表），UNION ALL 查询会扫描所有表
3. 查询时间范围如 `time_filter=1h` 但仍扫描所有分表，不必要
4. 按月分表的命名规则已知，可根据 `since` 参数推算只需查询的分表

**优化建议**：

```go
// 根据时间范围确定需要查询的分表
func getRelevantTables(db *gorm.DB, since time.Time) []string {
    var tables []string
    now := time.Now()
    t := time.Date(since.Year(), since.Month(), 1, 0, 0, 0, 0, since.Location())
    for t.Before(now) || t.Equal(now) {
        tableName := GetTrafficTableName(t)
        if db.Migrator().HasTable(tableName) {
            tables = append(tables, tableName)
        }
        t = t.AddDate(0, 1, 0)
    }
    return tables
}
```

### 4.5 🟢 低风险：API Key 查找使用线性 Map

**位置**：`cache/cache.go`

**问题分析**：API Key 查找使用 `map[string]*APIKeyInfo`，Go 的 map 本身是哈希表，查找效率 O(1)，当前设计合理。但在 API Key 数量极大时（如 10 万+），内存占用需关注。

**优化建议**：当前规模下无需优化。若未来 API Key 数量超过 10 万，可考虑使用 Bloom Filter 做预筛选。

### 4.6 🟢 低风险：前端开发代理配置泄露

**位置**：`web/vite.config.ts`

```typescript
server: {
    proxy: {
        '/api': {
            target: 'http://localhost:8080',
            changeOrigin: true,
        },
    },

```

**问题分析**：Vite 开发代理仅影响开发环境，不影响生产。但需确保生产环境的前端通过 Nginx 等反向代理转发 API 请求，而非直接暴露后端端口。

## 五、代码质量问题

### 5.1 缺少请求体校验

**位置**：`admin/admin.go` 第 547-569 行

```go
type CreateProviderReq struct {

    Description string `json:"description" binding:"max=500"`
    BaseURL     string `json:"base_url" binding:"required,url,max=500"`
    APIKey      string `json:"api_key" binding:"required,max=500"`
    Timeout     int    `json:"timeout" binding:"min=1,max=300"`
    Retry       int    `json:"retry" binding:"min=0,max=10"`
}
```

### 5.2 数据库操作缺少事务

**位置**：`admin/admin.go` 第 590-601 行

```go
func (s *Server) handleDeleteProvider(c *gin.Context) {
    s.db.Delete(&database.Provider{}, id)           // 操作1
    s.db.Where("provider_id = ?", id).Delete(&database.ModelProvider{})  // 操作2
}
```

**问题分析**：删除供应商和相关映射不在同一事务中，如果操作2失败，供应商已删除但映射残留，导致数据不一致。`handleDeleteUser` 也有同样问题。

**优化建议**：

```go
func (s *Server) handleDeleteProvider(c *gin.Context) {
    id := c.Param("id")
    err := s.db.Transaction(func(tx *gorm.DB) error {
        if err := tx.Delete(&database.Provider{}, id).Error; err != nil {
            return err
        }
        if err := tx.Where("provider_id = ?", id).Delete(&database.ModelProvider{}).Error; err != nil {
            return err
        }
        return nil
    })
    // 处理错误...
}
```

### 5.4 缺少优雅关闭

**位置**：`server/main.go`

```go
go func() {
    if err := proxyServer.Start(cfg.ProxyListen); err != nil {
        log.Fatalf("代理服务器启动失败: %v", err)
    }
}()
adminServer.Start(cfg.AdminListen)
```

**问题分析**：程序收到 SIGTERM 信号时直接退出，不会等待：

1. 正在处理的代理请求完成（可能导致客户端收到连接重置）
2. 流量记录器排空缓冲区（可能丢失最后一批流量记录）
3. 数据库连接正常关闭

**优化建议**：

```go
// 监听系统信号
quit := make(chan os.Signal, 1)
signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
<-quit

log.Println("正在关闭服务...")
ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
defer cancel()

proxyServer.Shutdown(ctx)
adminServer.Shutdown(ctx)
recorder.Stop()
```

### 5.5 前端路由守卫仅依赖客户端

**位置**：`web/src/router/index.ts`

```typescript
router.beforeEach((to, _from, next) => {
    const token = localStorage.getItem('token')
    if (to.meta.requiresAuth && !token) {
        next({ name: 'Login' })
    }
    // ...
})
```

**问题分析**：路由守卫仅检查 Token 是否存在，不验证其有效性。过期或伪造的 Token 仍可通过前端守卫，直到 API 请求返回 401 才被拒绝。

**优化建议**：在路由守卫中解析 JWT 的过期时间（无需密钥，仅需 Base64 解码 payload），提前拦截过期 Token。

## 七、架构改进建议

### 7.1 添加审计日志

所有管理操作（创建/删除用户、供应商、模型等）应记录审计日志，包含操作者、操作类型、操作对象、操作时间，便于安全事件追溯。

### 7.2 添加 Prometheus 指标

暴露 Prometheus 兼容的 metrics 端点，监控代理请求延迟、错误率、供应商健康状态、缓存命中率等关键指标。

### 7.3 支持 TLS

当前两个服务均监听 HTTP，生产环境应支持直接提供 TLS，或在部署文档中明确要求前置 Nginx 反向代理。

### 7.4 供应商健康检查

当前供应商状态需管理员手动更新，建议添加自动健康检查：定期向各供应商发送轻量请求（如模型列表），自动标记不可用的供应商为 `cooldown` 状态。

### 7.5 API Key 权限粒度

当前 API Key 仅关联用户，建议增加权限维度：允许单个 Key 限制可使用的模型、设置调用频率上限、设置过期时间。
